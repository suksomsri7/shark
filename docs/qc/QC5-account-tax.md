**สิ่งที่ตรวจแล้ว "ผ่าน"** (บันทึกไว้เพื่อไม่ให้ตรวจซ้ำ):
- ✅ เลขมัดจำ §F2: 1,070 (ฐาน 1,000 + VAT 70) / 5,350 (ฐาน 5,000 + VAT 350) → ใบกำกับใบที่สอง ฐาน 4,000 + VAT 280, VAT รวม 2 ใบ = 350, journal Σdr=Σcr = 528,000 สตางค์ทั้งสองฝั่ง — **เลขถูกทุกตัว**
- ✅ หลักการ VAT มัดจำ = accrual ตอนรับเงิน + ออกใบกำกับทันที — ถูกตาม ม.78(2)/78/1 (ความรับผิดเกิดเมื่อรับชำระราคาก่อนส่งมอบ/ก่อนให้บริการ)
- ✅ เลข §F5: 10,000 + VAT 700 = 10,700, WHT 5% ของฐาน 10,000 = 500, จ่ายจริง 10,200 — เลขถูก, WHT คิดจากฐานก่อน VAT ถูก (§11.2), หักตอน "จ่าย" ไม่ใช่ตอน issue ถูก
- ✅ ปัดเศษ VAT ระดับเอกสาร (bill-level) + แยกฐานต่อกลุ่มอัตราก่อนปัด + บรรทัดปรับเศษ ≤1 สตางค์ (§11.1) — แนวปฏิบัติที่สรรพากรยอมรับโดยทั่วไปคือคำนวณจากมูลค่ารวมในใบกำกับ (bill-level) → สเปคเลือกถูกทาง (ดู verify V2)
- ✅ รองรับ VAT 7% / 0% / ยกเว้น ต่อบรรทัด (`vatRateBp 700|0|-1`) + ผสมในใบเดียว
- ✅ ภาษีซื้อ "รอรับใบกำกับ" (1155) ห้ามเคลม จนกด RECEIVED จึงโอนเข้า 1150 + เพดานเคลมย้อน 6 เดือน + เตือนใกล้หมดสิทธิ์ (§3.2, §11.9) — ตรงกติกา ภ.พ.30
- ✅ WHT สองขาแยกชัด: ถูกหัก → สินทรัพย์ 1160 (เครดิตภาษี) / หักเขา → หนี้สิน 2130 + 50 ทวิ + ภ.ง.ด.3 (บุคคล)/53 (นิติ) แยกตาม LegalType — ถูก
- ✅ ใบลดหนี้/เพิ่มหนี้ บังคับอ้างเอกสารเดิม + มีผล VAT ในเดือนที่ออก CN/DN (โพสต์ตอน issue) — ถูกตาม ม.82/10
- ✅ เอกสาร immutable + void ไม่ reuse เลข + reversal journal — ตรงข้อกำหนดการจัดทำรายงาน/เอกสารของสรรพากร

| V8 | **รูปแบบการแสดงการหักมัดจำบนใบกำกับใบสุดท้าย** — แนวปฏิบัติออกใบกำกับ "มูลค่าคงเหลือหลังหักมัดจำ" เป็นที่ยอมรับทั่วไป แต่รูปแบบการอ้างใบกำกับมัดจำเดิมบนใบ (แสดงยอดเต็ม−มัดจำ หรือแสดงเฉพาะยอดคงเหลือ) ให้นักบัญชียืนยัน template | กระทบ PDF template ใบกำกับ + F2 |
| V9 | **การยกเลิกใบกำกับและออกใบใหม่ (REPLACE)** — ถ้อยคำหมายเหตุบังคับบนใบใหม่ ("เป็นการยกเลิกและออกใบกำกับภาษีใหม่แทนฉบับเดิม เลขที่... ลงวันที่...") และหน้าที่เก็บใบเดิมคู่สำเนา — ยืนยันถ้อยคำตามแนวปฏิบัติ (คำสั่งกรมสรรพากร ป.86/2542) แล้วบังคับใน template | กระทบ flow void+REPLACE ใน §3.1/S9 |

## ความเห็นรวมของผู้ตรวจ

**โครงหลักของสเปคนี้ "คิดแบบคนเข้าใจภาษีไทย"** — จุดที่ระบบบัญชี SME มักพลาด สเปคนี้ทำถูกหลายเรื่อง: VAT มัดจำเกิดตอนรับเงิน (เลขตัวอย่างถูกเป๊ะ), ภาษีซื้อรอใบกำกับแยก 1155 ห้ามเคลม, WHT คิดจากฐานก่อน VAT ตอนจ่าย, ภ.ง.ด.3/53 แยกตามสถานะนิติ, ใบลดหนี้บังคับอ้างใบเดิม, เอกสาร immutable + เลขไม่ reuse, double-entry ทั้งระบบ — ระดับนี้เกินค่าเฉลี่ยของสเปคระบบบัญชีที่ผู้ตรวจเคยเห็น

**แต่ ณ ร่างปัจจุบัน ยังปล่อยให้ SME ใช้ยื่นภาษีจริงไม่ได้** ด้วยเหตุ 3 กลุ่ม:

1. **จังหวะออกใบกำกับภาษี (C1-C3) เป็นช่องโหว่เชิงกฎหมายที่แท้จริง** — flow default พาไปออกใบกำกับตอนรับเงินสำหรับขายสินค้าเงินเชื่อ (ผิด tax point), บริการรับเงินหลายงวดออกใบกำกับงวดถัดไปไม่ได้ (constraint 1:1), และเดือนภาษีใน ภ.พ.30 มี 2 source ที่ขัดกันได้ — สามข้อนี้ทำให้ยอดยื่น ภ.พ.30 ผิดเดือน/ขาด ซึ่งคือเบี้ยปรับ-เงินเพิ่มจริงของผู้ใช้ ต้องแก้ก่อนเขียนโค้ด P1 เพราะกระทบ state machine และ data model (ใบกำกับผูก payment)
2. **เอกสารภาษียังไม่ "ครบตามแบบ"** — ใบกำกับเต็มรูป (ม.86/4 + เอกสารออกเป็นชุด), ใบกำกับอย่างย่อ (ภ.พ.06/ข้อความบังคับ), CN/DN (มูลค่าเดิม-ใหม่-ผลต่าง), 50 ทวิ (เงื่อนไขการหัก), รายงานภาษีขาย/ซื้อ (คอลัมน์สาขา), ภ.พ.30 (เครดิตยกมา) — สเปคใช้คำว่า "ครบตามสรรพากร" แต่ไม่ list ฟิลด์ ทำให้ implement แล้วหลุดได้ทุกจุด ควรแปลง findings M1-M8 เป็น checklist ฟิลด์ในสเปคเลย
3. **Guardrail สำหรับคนไม่รู้ภาษี** — กลุ่มเป้าหมายคือ SME ที่ไม่มีนักบัญชีใน loop รายวัน: ต้องบล็อกออกใบกำกับเมื่อไม่ได้จด VAT (C4), บล็อก/เตือนภาษีซื้อต้องห้าม (M9), เตือนกำหนดยื่น (m4) — ของพวกนี้ราคาถูกแต่กันเบี้ยปรับให้ผู้ใช้ได้มากที่สุด

**คำแนะนำเชิงลำดับ**: แก้ C1-C4 + M7 (ลำดับคำนวณ INCLUDE) ให้จบใน P1 เพราะฝังใน Document/state machine · M1-M6, M8-M9 แก้ในสเปค template/report ก่อนเฟสที่เกี่ยว (P1-P3) · รายการ V1-V9 ส่งนักบัญชี/ผู้สอบบัญชีภาษีอากรตรวจหนึ่งรอบก่อน freeze — โดยเฉพาะ V1 (checksum นิติบุคคล) ที่ถ้าผิดจะบล็อกผู้ใช้ตั้งแต่วันแรก

หลังแก้ครบตามนี้ ระบบจะอยู่ระดับ "ใช้เตรียมข้อมูลยื่นเอง + ส่งนักบัญชีตรวจ" ได้อย่างมั่นใจ ซึ่งตรงกับ positioning ระดับ FlowAccount/PEAK ที่สเปคตั้งไว้
