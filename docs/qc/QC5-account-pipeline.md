## ความเห็นรวม

สเปค 12-account เขียนดีมากในระดับ architecture (A1-A7 ชัด, mapping เมนู→docType→filter ครบ, posting rules ต่อชนิด, edge cases แหลม) — ปัญหาของสเปคมีเพียง 3 จุดที่ขัดกันเอง (M11 docNo, M12 unique payment, N4 receipt) กับ 1 ช่องนิยาม (M14 CONVERT.amount)

Implementation P1 วาง "โครงกระดูก" ตรงสเปคดี (ตารางเดียว polymorphic, relation, สตางค์ Int, immutable DRAFT-only, จองเลขใน tx) แต่ **ตัดหัวใจของสเปคออกสองเรื่องโดยพลการ**: (1) posting engine ที่สเปคสั่งให้มีตั้งแต่ P1 และ (2) กลไกที่ทำให้เอกสาร P1 หลายชนิด "ใช้จริงได้" (มัดจำ, วางบิล, ผลของ CN/DN) — บวกกับไม่มีสิทธิ์/audit เลย ทำให้สถานะจริงคือ *P1 ~60% ของ scope สเปค* ควรปิด C1-C4 + M3/M6 ก่อนรับเอกสารจริงจากผู้ใช้ เพราะเป็นกลุ่มเดียวที่ "ยิ่งช้ายิ่งแพง" — ที่เหลือ additive ตามหลัง P2 ได้โดยไม่รื้อ
