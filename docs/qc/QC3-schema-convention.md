### ผลตรวจรายเรื่อง (ที่ "ผ่าน" — ไม่เปิด finding)

- **② Scope compliance: ผ่าน** — โมดูล unit-scoped (01,02,03,04,05,14) ทุก model มี `tenantId + unitId` ครบ (รวมตารางลูกอย่าง `RestaurantOrderItemOption`, `HotelCancelRule`, `PosStockCountLine`) · โมดูล tenant (06,07,08,09,10,11,13) ไม่มี unitId เป็น isolation — ที่มีเป็น optional tag ตามอนุญาต (`MemberActivity.unitId?`, `PointLedger.unitId?`, `ChatConversation.unitId?`, `MeetingRoom.unitId?`, `KanbanBoard.unitId?`, `Reward.applicableUnitIds`, `RewardRedemption.fulfilledUnitId`) · `CouponRedemption.unitId` เป็น **required tag** เพื่อ attribution (ไม่ใช่ scope) — ถูกต้องตามเจตนา 08 · 12 hybrid ตามตกลง: `LedgerAccount` tenant-level, `AccountMapping.unitId?` (default/override), ตารางเอกสารทั้งหมด unit-scoped · 15 platform-level ตามออกแบบ (ดู m7)
- **③ Unique pattern: ผ่าน** — เลขเอกสาร/code ทุกตัวถูก scope: `HotelBooking [unitId,code]`, `Appointment [unitId,code]`, `TicketOrder [unitId,orderNo]`, `EventTicket [unitId,ticketNo]`, `PosSale [unitId,receiptNo]`, `JournalEntry/Expense/TaxInvoice [unitId,docNo]`, `QueueTicket [unitId,typeId,businessDate,seq]` · tenant: `CustomerProfile [tenantId,memberCode]`, `CouponCode [tenantId,code]`, `RewardRedemption [tenantId,code]`, `PointLedger [tenantId,idempotencyKey]` · unique ใต้ parent (เช่น `EventZone [eventId,name]`, `TicketAllocation [sessionId,ticketTypeId]`, `OptionChoice [groupId,name]`, `HotelCancelRule [ratePlanId,daysBefore]`) ถือว่าผ่านเพราะ parent เป็น unit-scoped อยู่แล้ว — แนะนำบันทึก pattern นี้ใน _CONVENTIONS กัน reviewer ทักผิด · จุดที่พึ่ง Postgres NULL semantics มีบันทึกกำกับครบ (06 `[tenantId,userId]`, 12 `AccountMapping` §11.9, REPORTS DailyStat)
- **⑤ Permissions matrix: ครบทุกไฟล์** — 01–14 มีตาราง OWNER/MANAGER/STAFF ครบ (15 ใช้ SUPER_ADMIN/SUPPORT/FINANCE ซึ่งถูกต้องสำหรับ platform) · การตีความ MANAGER สอดคล้องกัน: โมดูล unit = เต็มสิทธิ์ใน unit ของตน ✓ (01 §9, 02 §9, 03 §9, 04 §9, 05 §9, 12 §9, 14 §9 ระบุตรงกัน) · โมดูล tenant จำกัด MANAGER ในเรื่อง "นโยบายเงิน/องค์กร" (reward.write, coupon.write/issue, point.rule/settings, member.tier/settings, account.chart/period.close = OWNER) — **ความเห็น QC: ถูกทิศ** (MANAGER คุม "หน่วย" ไม่ใช่นโยบาย tenant) แต่ principle นี้ควรเขียนใน BLUEPRINT (ดู m11) · ข้อยกเว้นเดียวที่ต้องบันทึก = Kanban (m11)
- **⑦ เงิน Int สตางค์ + immutable: ผ่าน** — ทุกโมดูลที่มีเงินใช้ `Int` สตางค์ + เปอร์เซ็นต์เป็น basis points Int (`discountPercentBp`, `vatRateBp`, `occupancyBps`, `serviceChargeBps`) · `Decimal` ที่พบเป็นตัวคูณ/ปริมาณ ไม่ใช่เงิน (`MemberTier.pointMultiplier`, `PointRule.multiplier` Decimal(4,2), `Ingredient.*` ปริมาณวัตถุดิบ) — อนุโลมได้, `PlatformDailyStat.gmvSatang BigInt` เหมาะสมสำหรับ aggregate · เอกสารเงิน immutable ครบ: Hotel folio (void+reversal), PosSale (void/refund = เอกสารใหม่), JournalEntry (reversal), TaxInvoice (void+replaces), PlatformInvoice/ใบเสร็จ (void ก่อนจ่าย, ใบเสร็จห้ามแก้), Redemption จบสถานะห้ามแก้ — ข้อยกเว้นที่ประกาศชัด: `PosSale.pointEarned` (ผล async), `PointLedger.lotRemaining` (bookkeeping) — ยอมรับได้เพราะไม่ใช่มูลค่าเอกสาร
