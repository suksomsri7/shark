# 11 · Disaster Recovery (DR) Runbook

> กู้ข้อมูลเมื่อเกิดเหตุ: ลบผิด, migration พัง, ข้อมูลเสียหาย, หรือ purge PDPA พลาด (WO-0042)
> ฐานข้อมูล = **Neon Postgres** (ap-southeast-1) · ทุกขั้นตอนต้องซ้อมจริงก่อนถือว่าใช้ได้

## 1. สิ่งที่ Neon ให้มาฟรี (ต้องรู้ก่อนเหตุเกิด)
- **PITR (Point-in-Time Restore)** — ย้อน DB กลับไปวินาทีใดก็ได้ภายใน retention window (แผนปัจจุบัน ~7 วัน; ตรวจใน Neon Console → Project → Settings → Storage)
- **Branch** — คัดลอก DB ทั้งก้อน ณ เวลาหนึ่ง copy-on-write (ไม่คิดพื้นที่ซ้ำ) → กู้แบบ "เปิด branch แล้วดูดข้อมูลคืน" โดยไม่แตะ production
- Retention คือเพดานการย้อน — **เกินหน้าต่างนี้ = กู้ไม่ได้** จึงต้องซ้อม + สำรอง logical dump เป็นระยะ

## 2. Playbook A — ลบข้อมูลผิด / purge PDPA พลาด (ต้องกู้ 1 ร้าน ไม่แตะร้านอื่น)
เหตุ: `sweepPendingDeletes` ลบร้านที่ไม่ควรลบ หรือ query ผิดลบข้อมูลหาย

1. **หยุดเลือดก่อน** — ปิด cron ชั่วคราว: ถอด/ปิด `SHARK_CRON_SECRET` หรือ disable cron ใน `vercel.json` (กัน sweep รอบถัดไปซ้ำเติม)
2. หาเวลา T ก่อนเกิดเหตุ (จาก OpsEvent / log / เวลาที่ user แจ้ง)
3. Neon Console → **Branches → Create branch → "Restore from timestamp"** เลือกเวลา T-เผื่อ (เช่น T ลบ 5 นาที) ตั้งชื่อ `dr-<เหตุ>-<วันที่>`
4. ต่อ branch ใหม่ด้วย connection string ของมัน (อย่าเพิ่งชี้ production มา) แล้ว **ดูดเฉพาะแถวของร้านนั้น** กลับเข้า production:
   - export จาก branch: `pg_dump --data-only --table=... ` หรือ `SELECT ... WHERE "tenantId" = '<id>'` → COPY
   - หรือใช้ `exportTenantData(tenantId)` (src/lib/platform/pdpa.ts) ชี้ branch เพื่อดึง JSON ต่อ tenant แล้ว replay
   - **กรอง tenantId เสมอ** — ห้ามดูดทั้ง DB ทับ production (จะทับข้อมูลใหม่ของร้านอื่น)
5. ตรวจนับแถวเทียบก่อน/หลัง → เปิด cron กลับ

## 3. Playbook B — migration/deploy พัง ทั้ง DB เสียหาย
1. หยุด traffic (Vercel: rollback deployment เป็นตัวก่อนหน้า)
2. เลือกทางใดทางหนึ่ง:
   - **PITR reset** — Neon Console → Restore → ย้อน production branch กลับเวลา T ก่อน migration (กระทบทั้ง DB; เหมาะเมื่อเสียหายเป็นวงกว้าง)
   - **Branch แล้วสลับ** — สร้าง branch ณ T ที่ดี → verify → promote/ชี้แอปมา branch นั้น
3. ยืนยัน schema ตรงกับโค้ดที่ deploy (Prisma migrate status) ก่อนเปิด traffic

## 4. ต้องซ้อม (ไม่ซ้อม = ยังไม่มี DR)
- [ ] สร้าง branch จาก timestamp แล้วต่อได้จริง (จับเวลาที่ใช้ = RTO จริง)
- [ ] กู้ **1 ร้าน** กลับ production โดยร้านอื่นไม่กระทบ (ทดสอบ tenantId filter)
- [ ] ตรวจ retention window ปัจจุบัน + ตั้งเตือนถ้าใกล้เต็ม
- [ ] ซ้อม PITR reset บน branch ทิ้ง (ไม่แตะ production)
- [ ] เก็บ logical dump (`pg_dump`) เป็นระยะ สำหรับเหตุที่เกิน retention window

## 5. เบอร์/ค่าที่ต้องมีติดมือ
- Neon: Project ID + API key สร้าง/ลบ branch (ดู memory `reference_neon_shark`) — คนละตัวกับ `DATABASE_URL`
- `DATABASE_URL` (pooled runtime) · `DIRECT_URL` (migrate) — ดู `prisma.config.ts`
- Vercel: rollback deployment · cron toggle (`vercel.json` + `SHARK_CRON_SECRET`)
